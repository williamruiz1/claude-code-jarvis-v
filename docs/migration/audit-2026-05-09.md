## 1. The verdict in one paragraph

We rename the Claude Code Jarvis V app out of existence and absorb every one of its capabilities into VibeDispatcher as a new sub-surface called something like **"Terminal Voice"** (final name your call — see §6). VibeDispatcher's existing menu-bar shell, settings window, voice engine, and away-mode plumbing get extended to also cover the Claude Code CLI surface. The five Jarvis V pieces worth keeping (the Piper local Jarvis voice install, the multi-backend voice picker, the transcript pane, the hallucination filter, and the Keychain-stored API keys pattern) drop in as new modules under `vibedispatcher/voice/Sources/` and `vibedispatcher/menubar/Sources/`. The two pieces in Jarvis V that are redundant with what VibeDispatcher already does better (its own Speaker abstraction, its own TerminalMonitor) get dropped — Jarvis V's logic merges into VibeDispatcher's existing equivalents. Once the migration ships in VibeDispatcher v2.0, the standalone Jarvis V app is sunset. One repo, one brand, one menu-bar app, all four domains on `vibedispatcher.*` pointing at one product.

## 2. The two starting points (plain English)

### VibeDispatcher today
**What it is:** Your voice command center for VS Code, plus a phone app called Beacon that lets you check on your agents from anywhere.

- Lives in your menu bar
- Manages a queue of alerts from your VS Code Claude sessions; speaks them at you with a tunable cadence ("trickle mode")
- Listens for your voice via hotkey or a media-key press, transcribes it, figures out what you meant, runs the action
- Has a phone app (Beacon, served from a Fly.io server) so you can read events and send commands from your phone when away from the Mac
- 30,000 lines of code across Swift (menu bar + voice process), TypeScript (orchestrator + extension + relay + PWA)
- Currently on a feature branch (`feat/permissions-onboarding-2026-04-29`) — **not** main; needs to be reckoned with before any merge
- Already has its own TTS engine (`Speaker.swift`) supporting macOS `say`, AVSpeechSynthesizer, and ElevenLabs
- Already has its own Terminal monitor (`TerminalMonitor.swift`) — watches Terminal.app and iTerm2 for agent prompts, can type back into the right terminal

### Claude Code Jarvis V today
**What it is:** A small floating-widget app that gives Claude Code (the CLI tool you run in the terminal) a real two-way voice. Plays the actual JARVIS voice from Iron Man.

- Lives in your menu bar AND has a draggable HUD floating widget
- Wraps VoiceMode (an MCP server inside Claude Code) and adds visible structure around it: voice picker, transcript view, settings
- 5,700 lines of Swift, single-target SPM
- All 17 production-audit findings closed or locked
- Already at github.com/williamruiz1/claude-code-jarvis-v with proper Sparkle update infrastructure, EdDSA signing, code-signed release builds
- Distinctive things it has that VibeDispatcher does NOT have:
  - A ready, working install path for the Piper local JARVIS voice (model on disk, openedai-speech LaunchAgent serving on port 8001)
  - A proper voice picker UI with five backends (OpenAI, Kokoro local, Piper local, ElevenLabs, macOS native, Custom)
  - A purpose-built transcript pane with two render modes (minimal / chrome)
  - A hallucination filter for Whisper's "like and subscribe" hiccups
  - A clean MCP wrapper pattern that injects API keys from macOS Keychain at server-launch time

## 5. Clunky things I plan to remove (founder delegated this call)

### In VibeDispatcher itself

1. **Flat-file shared secret at `~/.vibedispatcher/secret`** — replaced by Jarvis V's KeychainHelper pattern. Closes the Regression Report C2 security finding.
2. **API keys in `config.yaml`** — also moved to Keychain via the same helper.
3. **Multiple Unix-socket daemons (core.sock, vscode-*.sock)** — keeping these for the VS Code surface where they're load-bearing, but the Terminal surface uses the lighter file-watch + JSONL-tail pattern from Jarvis V. **Don't extend the Unix-socket pattern to the new surface.**
4. **The 6-mode TricklePanel state machine (idle/playing/trickle/paused/returning/draft)** — keeps for VS Code (it's mature), but the new Terminal surface uses Jarvis V's simpler 2-state floating widget (idle/active). Two patterns coexist; users only see one at a time depending on which surface they're on.
5. **Bespoke Speaker abstraction with only 3 backends** — replaced by Jarvis V's VoiceCatalog with 5+ backends.
6. **No main window** — VibeDispatcher today has no main window, only the menu bar + floating panel. Jarvis V's MainWindowController template fills this gap.
7. **No code-signed Sparkle update channel** — Jarvis V's Sparkle infrastructure adapts in.

### In Claude Code Jarvis V (gets dropped during merge)

1. **Standalone AppDelegate** — merged into VD's
2. **Standalone build_app.sh + notarize_app.sh + build flow** — VD has its own (Xcode-based, more battle-tested)
3. **Standalone Sparkle key + appcast** — fresh keypair generated under VD's brand (Jarvis V's existing key gets retired with the archive)
4. **Standalone GitHub repo** — archived, not deleted

### The biggest clunk: two apps when one does it

Today, William has TWO menu-bar icons, TWO settings windows to learn, TWO sets of permissions to grant. After migration, ONE icon, ONE settings window with two tabs (VS Code · Terminal), ONE set of permissions.

## Addendum — VibeDispatcher WIP state (2026-05-09 inspection)

**The "branch reckoning" question simplifies.** I expected a feature branch with N commits ahead of main that needed land-first vs. rebase-into-migration. Reality is different:

- Branch `feat/permissions-onboarding-2026-04-29` exists locally only (no remote tracking)
- Branch HEAD == main HEAD (commit `66d81b0`) — **zero commits ahead of main**
- However, the working tree has substantial uncommitted work:
  - 18 modified files (most volume = `package-lock.json` churn; meaningful = 4 Swift files in `menubar/Sources/`)
  - 4 untracked files (~1140 lines new): `PermissionsManager.swift` (208), `PermissionsOnboardingView.swift` (269), `docs/permissions-onboarding-spec.md` (477), `docs/release-setup.md` (186)
  - Net Swift edits: AppDelegate +80, OnboardingWindowController +224, SettingsWindow +139, TricklePanelController +2 (333 lines)

**The collision risk.** Migration Phase 3 lands the Terminal surface and touches the same AppDelegate/SettingsWindow files. If we layer migration edits on top of uncommitted permissions-onboarding work, we get exactly the edit-revert thrash pattern that bit Plinthkeep (per the standing rule about not dispatching agents into repos with uncommitted founder WIP).

**The three options (your call):**

1. **You commit the WIP first.** Open a quick session, commit + push the permissions-onboarding work as PR #N. I rebase on top of that landed PR for migration Phase 1+. Cleanest separation. Recommended.
2. **You stash the WIP.** `git stash -u` puts everything aside; I migrate against clean main; you `git stash pop` after migration to deal with permissions-onboarding fresh. Risk: stash conflicts on pop.
3. **I work around the WIP.** Phases 1+2 don't touch AppDelegate/SettingsWindow — only substrate + Speaker. I can do those clean. Phase 3 must wait for the WIP to land. Lowest founder lift, but Phase 3 blocks until you commit.

**My pick: Option 3 for momentum.** I can start Phase 1 immediately (substrate extraction touches no Swift files in your WIP path — only adds new files under `voice/Sources/Substrate/`). Phase 2 also doesn't collide (Speaker.swift isn't in your WIP). Phase 3 needs the WIP resolved before it can dispatch. By the time Phase 3 is ready, you'll have had the natural break to commit permissions-onboarding.

**If you'd rather Option 1**, tell me and I'll pause the migration until you've shipped the permissions-onboarding PR.
