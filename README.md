# VoiceMode Monitor

macOS menu-bar companion for [VoiceMode](https://github.com/mbailey/voicemode) running inside Claude Code. Tracks the system mic, lists Claude sessions, lets you pick a TTS voice, and shows transcripts.

**Status:** v0.2 — main window + settings + voice picker + Sparkle/notarization wiring.

## What it does

- Sits in the menu bar as a waveform icon. Outlined = idle. Red = some app is using your mic.
- **Floating widget**: small always-on-top panel with a status pill and a "Sessions" pull-down to start a voice conversation in any Claude Code tab.
- **Main window** (open via menu bar → "Open Main Window…", or the floating widget's pull-down):
  - Left sidebar: live list of Claude sessions, sortable by status (active → idle → unknown). Double-click a session to focus it and start voice.
  - Center: transcript host (placeholder; the Transcript module plugs in here).
  - Toolbar: + New voice conversation · Refresh · Settings.
  - Status bar: current voice config (e.g. "OpenAI · onyx voice").
- **Settings** (toolbar button or menu bar → "Settings…"):
  - **Voice**: pick OpenAI built-in (alloy / echo / fable / onyx / nova / shimmer), ElevenLabs (voice ID + Keychain-stored API key), or Custom (full URL/model/voice override).
  - **Hallucinations**: edit the JSON list of patterns the transcript view filters out.
  - **About**: app version + GitHub link + "Check for Updates…" (Sparkle).

## Build + install

Requires macOS 13+ and Xcode command-line tools.

**Release build (Developer ID, hardened runtime, notarizable):**
```bash
./build_app.sh
```

**Dev build (any cert, no hardening — faster local rebuilds):**
```bash
VOICEMODE_DEV_BUILD=1 ./build_app.sh
```

**Build without Sparkle** (e.g. offline / sandboxed CI):
```bash
VOICEMODE_NO_SPARKLE=1 swift build -c release
```
The `SparkleBridge` falls back to a no-op so "Check for Updates…" surfaces a friendly alert instead of crashing.

The script installs to `~/Applications/VoiceMode Monitor.app` and launches it.

## App icon

A waveform-on-charcoal icon is generated by `scripts/generate_icon.swift`. Re-run if you tweak the rendering:
```bash
swift scripts/generate_icon.swift
```
This emits `Resources/AppIcon.iconset/*.png` plus the assembled `Resources/AppIcon.icns`. `build_app.sh` copies the .icns into the bundle's Resources/.

## Voice configuration flow

The voice/model/backend selection lives in UserDefaults under `voicemode-monitor.tts.*`. When you change it, we write a shell-sourceable env file at:

```
~/Library/Application Support/VoiceModeMonitor/voicemode-env.sh
```

Your voicemode MCP wrapper command in `~/.claude.json` should source this file before exec'ing voicemode. Example wrapper:

```json
"voice-mode": {
  "command": "/bin/sh",
  "args": [
    "-c",
    ". \"$HOME/Library/Application Support/VoiceModeMonitor/voicemode-env.sh\" 2>/dev/null; export OPENAI_API_KEY=$(security find-generic-password -s openai-api-key -a $USER -w) && exec uvx voice-mode"
  ]
}
```

For ElevenLabs, the same wrapper additionally exports the ElevenLabs key from Keychain:

```json
"voice-mode": {
  "command": "/bin/sh",
  "args": [
    "-c",
    ". \"$HOME/Library/Application Support/VoiceModeMonitor/voicemode-env.sh\" 2>/dev/null; export OPENAI_API_KEY=$(security find-generic-password -s elevenlabs-api-key -a $USER -w) && exec uvx voice-mode"
  ]
}
```

> ElevenLabs voicemode reads its key from `OPENAI_API_KEY` because voicemode treats it as a generic OpenAI-compatible API. Don't be confused by the variable name.

**No Claude Code restart required.** Voice changes take effect on the **next** Claude session that boots voicemode. Already-running sessions keep their old voice until you `/exit` and re-launch claude in that tab.

### Storing the ElevenLabs API key

Settings → Voice → ElevenLabs → "Save Key" writes the key to Keychain at `service=elevenlabs-api-key, account=$USER`. Or do it manually:

```bash
security add-generic-password -s elevenlabs-api-key -a $USER -w "<your key>" -U
```

`security` is also the same CLI the wrapper uses to read it back, so the storage convention is unified.

## Hallucination filter

The Transcript module reads `~/Library/Application Support/VoiceModeMonitor/hallucination-patterns.json` — a simple JSON array of strings. Edit it from Settings → Hallucinations. Lines that exactly match any pattern are suppressed in the transcript view.

## Sparkle appcast publishing

The app is wired to Sparkle 2.x via SPM (see `Package.swift`). The integration points:

- `Resources/Info.plist` keys: `SUFeedURL`, `SUEnableAutomaticChecks`, `SUScheduledCheckInterval`, `SUPublicEDKey`.
- `SparkleBridge.swift` instantiates `SPUStandardUpdaterController` at app launch and exposes `checkForUpdates(sender:)` for menu/toolbar wiring.

To stand up a real update channel:

1. **Generate signing keys** (one-time):
   ```bash
   # Sparkle ships a `generate_keys` tool inside the framework's Resources.
   # If you didn't keep the keypair, generate a new one and update SUPublicEDKey
   # in Info.plist with the public half. Treat the private key like a credential —
   # store in Keychain, never commit.
   .build/checkouts/Sparkle/bin/generate_keys
   ```
2. **Build + sign + notarize a release** (see Code-sign + notarize below).
3. **Wrap as a DMG or zip** and host it at a stable URL (e.g. GitHub Releases).
4. **Generate the appcast** with Sparkle's `generate_appcast` tool against your release artifacts directory; sign it with the private key from step 1.
5. **Upload `appcast.xml`** to a public URL.
6. **Update `SUFeedURL` in Info.plist** to point at that URL and ship a new release with the corrected URL.

Until the appcast is live, `SUFeedURL` points at a placeholder URL. Sparkle will fail update checks gracefully — the user sees "no updates available" instead of a crash.

## Code-sign + notarize flow

The release build path of `build_app.sh` does the following automatically:

1. `swift build -c release`
2. Assembles `~/Applications/VoiceMode Monitor.app`
3. Copies `Sparkle.framework` into `Contents/Frameworks/`
4. Signs Sparkle's nested helpers (Updater.app, Downloader.xpc, Installer.xpc) and the framework with `--options runtime --timestamp`
5. Signs the host app with the same flags
6. Verifies the signature with `codesign --verify --deep --strict`

After that you run:

```bash
./notarize_app.sh
# or pass an explicit path:
./notarize_app.sh "/path/to/VoiceMode Monitor.app"
```

That script:

1. Re-checks hardened runtime + secure timestamp are present.
2. Zips the .app with `ditto`.
3. Submits to Apple via `xcrun notarytool submit --wait`.
4. On success, staples the ticket back to the .app.

### One-time notarytool credential setup

```bash
# 1. Create an app-specific password at https://appleid.apple.com/account/manage
# 2. Save it as a Keychain profile so notarytool reads it without a prompt:
xcrun notarytool store-credentials "VoiceModeNotary" \
    --apple-id "you@example.com" \
    --team-id "5RRHNS4ZZB" \
    --password "<app-specific password>"
```

The profile name `VoiceModeNotary` is what `notarize_app.sh` reads. Override with `VOICEMODE_NOTARY_PROFILE=<name>` if you named yours differently.

### Picking a specific Developer ID identity

If you have multiple Developer ID certs, pin the one you want by SHA-1 hash:

```bash
VOICEMODE_RELEASE_IDENTITY="80333CAD375FA2A9AA37429B5CDC9487D78BC9A4" ./build_app.sh
```

`security find-identity -v -p codesigning` lists all available certs.

## How the mic detection works

`MicMonitor.swift` polls every 500ms for any input audio device with `kAudioDevicePropertyDeviceIsRunningSomewhere == 1`. Read-only — this app never claims the mic itself, so it does NOT show up in System Settings → Privacy → Microphone.
